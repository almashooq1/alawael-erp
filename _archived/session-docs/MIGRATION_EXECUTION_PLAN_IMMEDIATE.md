## 🔧 النقاط الحرجة المطلوب الاهتمام بها

### النقطة 1: البيانات المفقودة أو التالفة
```
المشكلة: ماذا لو قاعدة البيانات القديمة فيها بيانات خاطئة؟
الحل:
├─ فحص البيانات قبل النقل
├─ حذف السجلات التالفة (مع توثيق)
├─ معالجة القيم المفقودة (NULLs)
└─ إضافة قيم افتراضية آمنة

مثال:
❌ مستخدم بدون email (حقل إلزامي)
✓ الحل: استخدم email مؤقت = "user_import_[id]@temp.system" ثم أطلب من المستخدم تحديثها
```

### النقطة 2: Field Type Conversion
```
المشكلة: البيانات بنوع مختلف في النظام الجديد
أمثلة:
├─ INT → UUID (للـ IDs)
├─ TEXT → ENUM (للحالات)
├─ VARCHAR → JSON (للتفاصيل)
├─ UNSIGNED INT → BOOLEAN (نعم/لا)
└─ TIMESTAMP → ISO 8601 DATE

الحل:
├─ كل conversion لها function منفصلة
├─ اختبر التحويل قبل الإطلاق
└─ وثق أي فقدان بيانات
```

### النقطة 3: Character Encoding
```
المشكلة: النصوص العربية تصير "???" بعد النقل
الحل:
├─ تأكد أن UTF-8 مفعل على الجانبين
├─ اختبر نص عربي قبل النقل
└─ إذا فشل: استخدم mysqldump بـ UTF-8 صريح
```

### النقطة 4: Performance أثناء النقل
```
المشكلة: النقل يأخذ وقت طويل جداً (أكثر من 30 دقيقة)
السبب المحتمل:
├─ ملايين الصفوف (بدون pagination)
├─ Indexes مفعلة أثناء النقل (بطيء جداً)
├─ Constraints مفعلة أثناء النقل (فحص في كل صف)

الحل:
├─ عطل الـ indexes قبل النقل
├─ عطل Foreign Keys قبل النقل
├─ نقل البيانات على دفعات (batches) - كل 1000 صف
├─ أعيد تفعيل الـ indexes بعد النقل
└─ أعيد تفعيل Foreign Keys بعد النقل
```

## 🆘 ماذا تفعل إذا حدثت مشكلة

### المشكلة: Script الهجرة توقف في المنتصف
```
الإجراء الفوري:
1. اضغط Ctrl+C لإيقاف Script
2. اتصل بـ Database Admin وأخبره عن رسالة الخطأ
3. تحقق من السجل (logs) لتحديد السبب
4. قرر: تابع أم تراجع؟

معالجة المشكلة:
├─ إذا كانت مشكلة بسيطة (مثل timeout): أعد المحاولة
├─ إذا كانت مشكلة معقدة: استدعِ Programming Lead
└─ إذا فشلت المحاولة الثانية: نفذ Rollback

الوقت المسموح: 15 دقيقة من البداية
يعني إذا استغرقت أكثر من 15 دقيقة بدون نتائج → Rollback
```

### المشكلة: البيانات خاطئة بعد النقل
```
مثال: عد الصفوف لا يتطابق

الساعة: 8:00 AM (نهاية النقل)
OLD DB: 1,523 users
NEW DB: 1,421 users (تنقص 102 user!)

الإجراء:
1. توقف الاختبار الآن 🛑
2. اتصل بـ Database Admin
3. لا تفعل rollback فوراً - أولاً…

التحقق:
├─ هل النقل اكتمل (أم توقف في المنتصف)؟
├─ هل هناك تحويل فشل (users بدون email)؟
├─ هل بيانات قديمة محذوفة في القديمة؟
└─ كم الوقت نحتاج لإصلاح (ساعة؟ دقائق؟)

إذا الإصلاح < 15 دقيقة → اصلح
إذا الإصلاح > 15 دقيقة → Rollback
```

### المشكلة: Rollback مطلوب
```
الساعة التي قررت نفذ Rollback: ⏰

SQL Commands:
$ mysql> DROP DATABASE alawael_new;
$ mysql> CREATE DATABASE alawael_new;
$ mysql> mysql alawael_new < backup_before_migration_FEB28_0600.sql;

النتيجة: مرجع النظام الجديد إلى حالة فارغة، والقديم يعود للخدمة

الوقت المتوقع: 10-15 دقيقة
```
