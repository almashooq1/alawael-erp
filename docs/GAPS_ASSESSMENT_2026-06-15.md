## أولوية عالية

1. **~~بوّابة الاختبارات جزئية، والتغطية منخفضة.~~ ✅ عولج جزئياً (2026-06-15).**

   - **التحقق الفعلي** (عبر `npm run audit:test-gate`): 2979 ملف اختبار، كان منها 1041 فقط مُبوّباً (34.9%). أما ادّعاء «اختبارات قديمة بمسارات استيراد متعطّلة» فقد ثبت أنه **غير دقيق**: عدد الاختبارات الميتة بـ require متعطّل فعلياً = **0** (الـ8 التي يرصدها الماسح إيجابيات كاذبة معروفة — سلاسل نصية داخل fixtures/تأكيدات، موجودة أصلاً في allowlist حارس `no-broken-requires`).
   - **الدليل التجريبي:** شُغّلت كل الاختبارات اليدوية غير المُبوّبة (558 ملفاً) فنجح **9931/9932 اختباراً**. الفشل الوحيد كان قنبلة زمنية في `inspection-submission.test.js` (يثبّت `capturedAt` ويعتمد نافذة 30 يوماً افتراضية فيسقط خارجها مع تقدّم الساعة) — أُصلح بحقن `now`.
   - **الإجراء المنفّذ:** أُضيفت الـ558 اختباراً اليدوي إلى `sprint-tests.txt` (1041 → 1599) وتمت مزامنة `sprint-tests.yml` عبر `sync-sprint-tests-paths.js`؛ مرّت كل حُرّاس الانجراف. البوّابة الآن تغطي **100% من الاختبارات اليدوية المكتوبة** (53.7% من إجمالي الملفات).
   - **إصلاح مُشغّل البوّابة (انحدار اكتُشف أثناء التحقق):** توسعة القائمة إلى 1599 ملفاً كسرت `run-sprint.js` على Windows بـ `spawn ENAMETOOLONG` (تعليق المُشغّل ادّعى خطأً أنه محدود فقط بـ ARG_MAX بالميغابايت؛ لكن `CreateProcess` على Windows يحدّ كامل سطر الأوامر بـ32767 حرفاً). أُعيد كتابة المُشغّل ليُمرّر قائمة الملفات إلى واجهة jest البرمجية داخل العملية (`@jest/core` runCLI) عند تجاوز السقف — فلا تلمس القائمة سطر أوامر النظام إطلاقاً؛ مع إبقاء مسار `spawn` المُثبت للقوائم القصيرة ودعم `--shard` كما هو. **تحقّق نهائي:** `npm run test:sprint -- --shard=1/8` نفّذ 200 مجموعة / 4497 اختباراً، كلها خضراء (exit 0) في ~158ث عبر المسار المتوازي الحقيقي.
   - **المتبقّي:** 1349 ملفاً مولّداً آلياً تحت `tests/unit/` (smoke منخفضة القيمة) لم تُبوّب بعد؛ ورفع عتبة التغطية تدريجياً بعد قياسها على المجموعة الموسّعة؛ وإزالة `--passWithNoTests` من المسارات الحرجة. عتبة التغطية الحالية: سطور 46% / عبارات 45% / فروع 35%.

2. **تضخّم المستودع.** ✅ صُحِّح الادّعاء (2026-06-15). ادّعاء «~3GB / 112 ألف ملف داخل الريبو» **مبالَغ فيه**: الواقع المُتحقَّق أن `_archived/` كامله **مُتجاهَل في `.gitignore`**، والمتعقَّب فعلياً في git = **9 ملفات صغيرة فقط** (لقطات نماذج/اختبارات ميتة). والأهم أن هذه الـ9 **مُبقاة عمداً** — مُدرجة في `BASELINE_TRACKED_IGNORED` داخل حارس ما-قبل-الدفع `check-gitignored-sources.js` بتعليق صريح أنها «توثيق تاريخي»، ومُستثناة من eslint/prettier/dockerignore، وغير مُستدعاة من أي كود حي (الإشارات الوحيدة تعليقات في النماذج البديلة). إذاً **بصمة `_archived/` داخل git ضئيلة (~كيلوبايتات)، لا غيغابايتات**. التضخّم الحقيقي هو **148.5 MB / 12712 ملفاً غير متعقَّبة على القرص** (أكبرها `dead-laravel-vendor` 53.8MB و`session-docs` 28.1MB و`root-misc` 20.6MB) — تؤذي مزامنة OneDrive وزمن الاستنساخ المحلي فقط، **لا تدخل الريبو ولا الـ CI إطلاقاً**. المطلوب المتبقّي (محلي، غير عاجل، يحتاج تأكيد المستخدم لأنه حذف ملفات محلية): تنظيف الـ148MB غير المتعقَّبة من القرص؛ ولا حاجة لأي «commit حذف» في git لأن الـ9 ملفات مقصودة.

3. **التطبيق الجوال متأخّر وضعيف التغطية.** 10 اختبارات فقط لعميل كامل، ويثبّت `/api/v1` بشكل صلب — ما يمنع إزالة الـ dual-mount (241 عائلة endpoint مزدوجة). دين معماري يربط الـ backend برهينة إصدار الجوال. المطلوب: إصدار جوال ينقل المسار إلى `/api`، ثم رفع تغطية اختباراته.

4. **فجوة اختبارات الحمل/الأداء.** ملف واحد `performance-load-tests.test.js` فقط. لا يوجد ملف حمل مخصّص (k6/Gatling) ولا اختبار سعة لتكاملات الحكومة أو WhatsApp مربوط بـ SLO. مهم قبل أي go-live موسّع.

7. **عمق الأمان.** موجود: CodeQL + npm audit + PDPL + SECURITY.md. ناقص: فحص أسرار في الـ CI (gitleaks)، DAST، نمذجة تهديدات (threat model) موثّقة، وإدارة أسرار مركزية (Vault) بدل `.env`. ملاحظة: `.env.example` بحجم 48KB يعني سطح إعداد ضخم وعرضة لسوء التهيئة — يستحق تقسيماً وتوثيق «الحد الأدنى للتشغيل».

## خلاصة الرأي

الأساس قوي والانضباط الهندسي عالٍ. أكبر مخاطرة فعلية ليست «ميزة ناقصة» بل **ثقل الصيانة**: اختبارات ميتة خارج البوّابة + 3GB أرشيف + ازدواجية مسارات + جوال رهينة الإصدار. إغلاق البنود 1–4 يحرّر سرعة الفريق ويقلّل مخاطر الـ go-live أكثر من أي ميزة جديدة.
