# 23 — Go-Live Checklist

> **Purpose**
>
> The Al-Awael ERP ships in a "feature-flag off, infrastructure ready"
> posture. This checklist is the single document an operator follows to
> flip the switches at go-live, in the right order, with the right
> verification steps. It indexes every relevant env var, every onboarding
> command, and every probe that proves the switch worked.

## 0. Pre-flight

- [ ] **Backups exist + are restorable.** Run
      `node backend/scripts/dr-verify.js` on the VPS and confirm `PASS`.
      A restore drill must succeed before we trust production data.
      → runbook: [19-dr-verification.md](19-dr-verification.md)
- [ ] **Backup encryption key generated + escrowed.** The key must be
      saved in a password manager AND on the VPS — losing it makes
      every encrypted archive unrecoverable.
      → runbook: same as above (encryption section)
- [ ] **Ops-alerter recipients configured.** All of the alerting work
      below assumes `OPS_ALERT_EMAIL` and `OPS_ALERT_PHONE` are set.
- [ ] **VPS secrets in GitHub Environment.** `VPS_HOST`, `VPS_USER`,
      `VPS_SSH_KEY`. Verified by triggering `dr-verify.yml` workflow
      manually and watching it succeed.

## 2. NPHIES (insurance claims)

NPHIES integration starts in **mock mode** by default. Don't touch
these until the integration team has finished onboarding with CCHI.

```env
# Mock mode — safe defaults
NPHIES_MODE=mock

# Live mode (after CCHI onboarding)
# NPHIES_MODE=live
# NPHIES_BASE_URL=<HSB endpoint>
# NPHIES_CLIENT_ID=<CCHI provided>
# NPHIES_CLIENT_SECRET=<CCHI provided>
# NPHIES_PROVIDER_ID=<HPO license>
# NPHIES_RECON_ENABLED=true                # default; sweeps PENDING_REVIEW

# Webhook HMAC (after webhook URL is whitelisted with NPHIES)
# NPHIES_WEBHOOK_SECRET=<shared secret>
```

Verify:

- [ ] `curl /api/v1/webhooks/nphies` responds 405 (method not allowed) —
      route is mounted (post-deploy smoke catches this)
- [ ] Send a sample eligibility check via `POST /api/nphies/eligibility/check`
- [ ] First claim flow: `POST /api/v1/sessions/admin/:id/create-claim`
      then `POST /api/admin/nphies-claims/:id/submit`
- [ ] Tariff seed applied: `node backend/scripts/seed-insurance-tariffs.js`
- [ ] Insurance Tariffs admin page reachable at `/insurance-tariffs`
- [ ] Bulk dialog reachable from "إدارة الجلسات العلاجية" → "مطالبات مجمّعة"

## 3. ZATCA Phase 2 (e-invoicing)

ZATCA also defaults to mock. The wiring (post-save hook + SLA sweeper +
real-time alerts) lights up when the flags below flip together.

```env
# Mock mode
ZATCA_SIGNER_MODE=mock
FATOORA_MODE=mock

# Live mode (after onboarding each branch)
# ZATCA_SIGNER_MODE=live
# FATOORA_MODE=live
# FATOORA_BASE_URL=https://gw-fatoora.zatca.gov.sa/e-invoicing/core
# FATOORA_BINARY_TOKEN=<from ZATCA portal>
# FATOORA_RL_CAPACITY=600
# FATOORA_RL_REFILL_PER_MIN=600
#
# ZATCA_AUTOSUBMIT=true                     # auto-submit on Invoice.save
# ZATCA_SLA_SWEEPER_ENABLED=true            # 24h SLA enforcement
# ZATCA_SLA_INTERVAL_MS=1800000             # 30 min cadence
# ZATCA_SLA_WARN_MS=64800000                # 18h → retry
# ZATCA_SLA_BREACH_MS=82800000              # 23h → ops-alert
```

Onboarding sequence per branch (from the admin UI):

1. Open `/zatca-credentials` page
2. Click "إضافة بيانات اعتماد" — fill org name, VAT, CR, address
3. On the row, click the cloud-upload icon → "تسجيل لدى ZATCA"
4. Enter the OTP from ZATCA portal
5. Once status shows "مهيّأ" + "تجريبي", click the play icon →
   "ترقية إلى Production"
6. Status flips to "إنتاج" — the credential is ready

Verify:

- [ ] `GET /api/zatca-phase2/status` returns 200 with `mode` matching
      the env (post-deploy smoke covers this)
- [ ] `GET /api/admin/zatca-credentials` returns the seeded rows; all
      sensitive fields show `[REDACTED]` not their real values
- [ ] First invoice fires the post-save hook (check
      `zatca.zatcaStatus` after saving an Invoice with
      `ZATCA_AUTOSUBMIT=true`)
- [ ] Force a REJECTED response (mock mode supports this) — confirm
      ops-alerter fires immediately
- [ ] Wait 23h on a non-submitted simplified invoice — confirm SLA
      sweeper fires the aggregated alert

→ runbook: [22-zatca-phase2.md](22-zatca-phase2.md)

## 6. Schedulers in production

Three schedulers boot from `app.js` based on env flags:

| Flag                                  | Scheduler                      | Cadence |
| ------------------------------------- | ------------------------------ | ------- |
| `ENABLE_AUTO_BACKUP=true`             | Mongo backup + retention sweep | 24h     |
| `NPHIES_RECON_ENABLED=true` (default) | NPHIES reconciliation sweeper  | 10 min  |
| `ZATCA_SLA_SWEEPER_ENABLED=true`      | ZATCA B2C 24h SLA sweeper      | 30 min  |

Verify (after `pm2 restart all --update-env`):

- [ ] `pm2 logs alawael-api --lines 100` shows "[backup]", "[nphies-recon]", and "[zatca-sla]" startup lines
- [ ] After the first tick, each scheduler logs counts (or "scanned=0"
      when nothing's there yet)

## 7. Common gotchas

- **`pm2 restart` without `--update-env`.** Env changes don't take
  effect — flags stay off even though `.env` was edited.
- **`VPS_USER=root` but pm2 runs as alawael.** Use `sudo -u alawael
pm2 ...` for every command. The deploy workflow already does this;
  manual sessions are the trap.
- **Removing fields from `.env.example`.** It's the documentation
  source of truth. Add new flags here too — operators read this file.
- **Schedulers double-booting.** If you run a one-off script that
  imports `app.js`, the boot block fires. Use `--require` or extract
  the route registration into a separate module to avoid this.
- **OPS*ALERT*\* unset.** All alerting becomes silent log entries. The
  ops-alerter logs a warning when it has no recipients but doesn't
  refuse to be invoked.

## 8. Roll-back plan

If a deploy breaks production:

1. Identify the prior good commit:
   `gh pr list --state merged --base main` then click into one with a
   green deploy run.
2. SSH to VPS and check out that commit:
   ```bash
   cd /home/alawael/app
   sudo -u alawael git fetch && sudo -u alawael git checkout <sha>
   sudo -u alawael pm2 restart all --update-env
   ```
3. If even pm2 won't restart cleanly, the docker fallback at port 3001
   may still serve from an older image. Switch nginx to it temporarily:
   `sudo nginx -s reload` after editing the upstream block.

→ runbook: project memory `project_vps_deploy_stuck_2026-04-29.md`
has the full diagnostic for "deploy stuck behind by N commits".
