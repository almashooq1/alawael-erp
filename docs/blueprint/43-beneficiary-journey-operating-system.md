## ملخص تنفيذي (Executive Summary)

نظام الأوائل وصل إلى **المستوى 2 من النضج**: "ذاكرة موحّدة" من الطراز الأول — كل حدث سريري يتدفّق إلى سجل زمني واحد لكل مستفيد (`CareTimeline`)، وكل واجهة تقرأ منه. هذا إنجاز معماري حقيقي.

الفجوة بين ما هو قائم وما هو "عالمي المستوى" ليست في عدد الوحدات — بل في **ربط البيانات على مستوى النموذج** (الخيط الذهبي) وفي **طبقة الذكاء** التي تقرأ هذا الخيط. هذه الوثيقة:

1. تُعرّف **الرحلة المثالية** بثماني مراحل ببوابات جودة.
2. تُشخّص **ست فجوات في الخيط الذهبي** (مبنية على فحص الكود الفعلي) مع عقود البيانات لإغلاقها.
3. تصمّم **طبقة دعم القرار السريري (CDSS)** فوق الخيط الذهبي.
4. تضيف الطبقات الاحترافية العميقة: **أنطولوجيا الأهداف، حوكمة الذكاء، نموذج القيمة الاقتصادية، إدارة التغيير، المعايرة الدولية، والأصل البحثي**.
5. تضع **خارطة بناء (R1–R7)** قابلة للتنفيذ على نمط الوحدات (Waves).

**التوصيات الثلاث الأعلى أثراً:** (1) إغلاق فجوتي الهدف↔المقياس والجلسة↔الهدف؛ (2) فرض الإلزام في الواجهة؛ (3) تفعيل حزم المسارات حسب نوع الإعاقة.

# الجزء I — الأساس الاستراتيجي

## 1.1 الأطر الحاكمة الخمسة

| الإطار                               | ما يفرضه على الرحلة                                                        | الوحدة في النظام                                  |
| ------------------------------------ | -------------------------------------------------------------------------- | ------------------------------------------------- |
| **ICF** (التصنيف الدولي للأداء، WHO) | وصف الحالة بأربعة محاور: وظائف الجسم / النشاط / المشاركة / العوامل البيئية | `icf-assessments` · `Goal.icfMapping` (W452)      |
| **CBAHI** (الاعتماد السعودي)         | توثيق إلزامي لكل مرحلة + بوابات جودة                                       | 45 معياراً عبر `cbahi-standards.registry.js`      |
| **CRPD** (اتفاقية حقوق ذوي الإعاقة)  | صوت المستفيد + استقلاليته في القرار (لا استبدال)                           | `voice-log` · `decision-rights` · `self-advocacy` |
| **PDPL** (حماية البيانات الشخصية)    | موافقات صريحة + TTL ≥30 يوم + تدرّج حساسية                                 | `Consent` · sensitivity-grade lib                 |
| **EBP + Value-Based Care**           | كل تدخّل بدليل، وكل دليل بمخرَج قابل للقياس                                | الخيط الذهبي + MCID/GAS                           |

## 1.2 المبادئ المعمارية الأربعة (Design Tenets)

1. **سجل واحد للحقيقة** (Single Source of Truth) — النواة الموحّدة هي المرجع، لا الوحدات المتفرّقة.
2. **القياس قبل التدخّل** (No Goal Without a Measure) — لا هدف بلا خط أساس وأداة قياس وقيمة مستهدفة.
3. **النسبة لا التعليق** (Everything Belongs to Something) — كل جلسة لهدف، كل هدف لتقييم، كل تقييم لحلقة رعاية.
4. **الذكاء يقترح، الإنسان يقرّر** (Augment, Don't Automate) — النظام يرفع التوصية بدورة حياة، والأخصائي يعتمدها.

> هذه المبادئ الأربعة هي محكّ كل قرار تصميمي لاحق في الوثيقة. أي ميزة تخالف أحدها تُراجَع.

# الجزء II — نظام التشغيل السريري

## 2.1 دورة حياة المستفيد كآلة حالات رسمية

```text
[REFERRED] ──► [INTAKE] ──► [SCREENING] ──► [EPISODE_OPEN]
                                                  │
                                                  ▼
                                          [ASSESSMENT] ──► [PLANNING]
                                                  ▲              │
                                                  │              ▼
                  [DISCHARGED] ◄── [TRANSITION] ◄── [ACTIVE_THERAPY] ⇄ [REVIEW]
                       │                                   ▲   │
                       │             [ON_HOLD] / [ESCALATED]┘   │
                       └──────────────── [FOLLOW_UP] ◄─────────┘
```

- كل انتقال يولّد حدث نواة (`CareTimeline`) + يفرض بوابة CBAHI ذات صلة.
- `ON_HOLD` / `ESCALATED` فروع جانبية، لا حالات نهائية.
- الأساس القائم: `BeneficiaryLifecyclePanel` + episode state machine (ADR-031).

## 2.2 المراحل الثماني ببواباتها

| ------------------------------- | --------------------------------------------------------------- | ------------------------ | --------------------------------------------------------------- | --------------------- |
| **0 القبول**                    | فئة الإعاقة + ولي الأمر + **موافقات PDPL** + وثائق + إسناد فرع  | قبول/اجتماعي             | `beneficiaries/new` · `documents` · `Consent`                   | زمن القبول            |

## 2.3 تصنيف الحِدّة (Acuity Tiering)

ليس كل مستفيد يحتاج نفس الكثافة. يُصنَّف عند الفرز ويُعاد تقييمه كل مراجعة:

| المستوى                     | الوصف                                     | الكثافة                   | إعادة القياس   |
| --------------------------- | ----------------------------------------- | ------------------------- | -------------- |
| **T1 — معقّد/عالي الخطورة** | تعدد إعاقات · رايات سلامة نشطة · سلوك خطر | مكثّف · MDT شهري          | كل 4–6 أسابيع  |
| **T2 — متوسط**              | إعاقة واحدة رئيسية · أهداف نشطة           | معياري · MDT ربع سنوي     | كل 8–12 أسبوع  |
| **T3 — صيانة/استقرار**      | بلغ هضبة · الهدف الحفاظ                   | متباعد · مراجعة نصف سنوية | كل 12–24 أسبوع |

الحِدّة حقل على المستفيد يغذّي **محرّك توزيع الحالات** (`Caseload Matcher`).

# الجزء III — معمارية الخيط الذهبي (نموذج البيانات)

## 3.1 طبقتان معماريتان — التمييز الحاسم

| الطبقة                                        | الوصف                                                                                        | الحالة               |
| --------------------------------------------- | -------------------------------------------------------------------------------------------- | -------------------- |
| **(أ) النواة الموحّدة** (`CareTimeline`)      | سجل زمني لكل مستفيد، تتدفّق إليه كل الأحداث عبر `dddCrossModuleSubscribers.js` (60+ نوع حدث) | ✅ **مكتملة وقوية**  |
| **(ب) الخيط الذهبي** (روابط refs بين النماذج) | تقييم→هدف→جلسة→مقياس مترابطة بمفاتيح أجنبية حقيقية                                           | ⚠️ **مكتملة جزئياً** |

**الفرق الجوهري:** النواة تسجّل _أن_ جلسة اكتملت؛ الخيط الذهبي يعرف _أي هدف_ خدمت و*أي مقياس* سيتتبّعه. النواة = الذاكرة؛ الخيط الذهبي = العقل التحليلي.

## 3.2 النموذج المرجعي (Entity-Relationship)

```text
Beneficiary ──1:N──► EpisodeOfCare ──1:N──► CareTimeline (✅ النواة)
     │                     │
     │                     ├──1:N──► Assessment ──(goalId ⚠️ضعيف)──► Goal
     │                     │
     ├──1:N──► CarePlan ──1:N──► Goal ◄══[#1: linkedMeasures]══► MeasurementMaster
     │            (#4 refs)        │                                    │
     │                       (#2: goalIds)                         (defines)
     │                             ▼                                    ▼
     ├──1:N──► Session ════════════╝                      MeasurementResult
     │                                          (baseline #1 ──[#5: baselineResultId]──► progress)
     └──1:N──► GasScoring ──(goalId ✅)──► Goal
```

## 3.3 سجل الفجوات الست (مبني على فحص الكود الفعلي)

| #     | الرابط               | الحالة الآن                                                                                                                                                                                                                                | الإصلاح (تغيير نموذج)                                                                                                | السؤال الذي يفعّله               |
| ----- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------- | -------------------------------- |
| **1** | الهدف ↔ المقياس     | ✅ **مُنفَّذ (W1090)**                                                                                                                                                                                                                     | `Goal.linkedMeasures:[{measureId: ref MeasurementMaster, ...}]`                                                      | "بأي مقياس أُثبت تحقّق الهدف؟"   |
| **2** | الجلسة ↔ الهدف      | ✅ **مُنفَّذ (W1149)** — تبيّن بالتدقيق أن الرابط الأمامي موجود فعلاً ك‍ـ`ClinicalSession.goalProgress[].goalId` (ref `TherapeuticGoal`)، أغنى من `goalIds[]` مسطّح                                                                        | لا تُضِف `goalIds[]` مكرَّراً؛ W1149 ثبّته بحارس + فهرس عكسي `goalProgress.goalId` + أصلح خطاف الحفظ القديم          | "كم جلسة خدمت الهدف X؟"          |
| **3** | التقييم ↔ الهدف     | ✅ **مُنفَّذ (W1151)** — تبيّن بالتدقيق أن تدفّق W568 كان _يُسقط_ `applicationId` (يبقى أثره في tags/notes فقط)                                                                                                                            | `TherapeuticGoal.measureApplicationId` (ref MeasureApplication، sparse+index) + كتابته في `createGoalFromSuggestion` | "ما التقييم الذي وَلّد الهدف؟"   |
| **4** | الخطة ↔ الأهداف     | ◑ **جزئياً (W1154)** — الحافتان القانونيتان موجودتان: `TherapeuticGoal.carePlanId → UnifiedCarePlan` + جسر `Goal.therapeuticGoalId → TherapeuticGoal` (W1133)؛ W1154 قفلهما + فهرسهما. الـsubdocs المضمّنة القديمة تُوحَّد عبر ADR-026/040 | الحافتان مُقفلتان+مفهرستان؛ **تحويل subdocs→refs = قرار ADR-026/040** (مملوك)                                        | "أي خطط تضم الهدف X؟"            |
| **5** | خط الأساس ↔ التقدّم | ✅ **مُنفَّذ (W1145)**                                                                                                                                                                                                                     | `MeasurementResult.baselineResultId` + `isBaseline` + invariants                                                     | "كل قياسات التقدّم مقابل أساس X" |
| **6** | التقييم ↔ المقياس   | ◑ **جزئياً (W1153)** — الجسر القانوني موجود فعلاً ك‍ـ`MeasureApplication.assessmentId` (ref ClinicalAssessment)؛ W1153 ثبّته بحارس + فهرس عكسي. `MeasurementResult` القديم لم يُمدَّد (يُسيَّج عبر ADR-041)                                | الجسر القانوني مُقفل + مفهرس؛ **توحيد النظامين = قرار ADR-041/038** (مملوك)                                          | توحيد نظامَي التقييم             |

# الجزء IV — طبقة الذكاء (Clinical Decision Support System)

أربعة محرّكات تقرأ من الخيط الذهبي وتُنتج توصيات — كلها لها أساس قائم في النظام.

## 4.1 المحرّكات الأربعة

| المحرّك                    | المُدخَل                   | المُخرَج                        | أساسه القائم                                      |
| -------------------------- | -------------------------- | ------------------------------- | ------------------------------------------------- |
| **تنبؤ المخرجات**          | اتجاه المقاييس + خط الأساس | مسار متوقّع + احتمال بلوغ الهدف | `Outcome Forecasting` (W479/W248)                 |
| **كشف التدهور/الهضبة**     | غياب تحسّن عبر N قياسات    | تنبيه + توصية مراجعة خطة        | `Plateau/Regression adapters` (W337)              |
| **التنبؤ بالتصعيد**        | إشارات سلوك + حضور + سلامة | درجة خطر + تصعيد استباقي        | `Escalation Predictor` (5 sources, W433/W434)     |
| **التوصية السريرية (RAG)** | الحالة + الأدلة/السياسات   | اقتراح أهداف/تدخّلات مُستشهَد   | `RAG` + `AI recommendation lifecycle` (W283/W334) |

## 4.2 محرّك "الإجراء الأفضل التالي" (Next-Best-Action) — الطبقة الموحِّدة

يقرأ **حالة المستفيد** (آلة الحالات §2.1) + **حالة الخيط الذهبي** ويقترح الخطوة:

| الحالة           | الإشارة                        | الإجراء المقترح                    |
| ---------------- | ------------------------------ | ---------------------------------- |
| `ASSESSMENT`     | مضى 14 يوماً بلا خط أساس مكتمل | ⏰ "أكمل مقياس X لإغلاق التقييم"   |
| `ACTIVE_THERAPY` | جلسات بلا هدف منسوب            | 🔗 "اربط الجلسات بأهداف نشطة"      |
| `REVIEW`         | MCID فوق العتبة في هدف         | ✅ "اقترح إغلاق الهدف + هدف تالٍ"  |
| `ACTIVE_THERAPY` | هضبة عبر 3 قياسات              | 🔄 "راجع الخطة — تدخّل بديل مقترح" |
| أي حالة          | راية سلامة حمراء نشطة          | 🚨 "تصعيد فوري + إشعار MDT"        |

كل توصية تمر بدورة حياة `AI recommendation` (DRAFT → PENDING_REVIEW → APPROVED/REJECTED) — تطبيقاً للمبدأ 1.2.4.

## 4.3 لوحة "الانتباه الذكية" (Smart Attention Queue)

تحوّل من _pull_ (الأخصائي يبحث عن المشاكل) إلى _push_ (النظام يدفعها مرتّبة بالأولوية). الأساس القائم: `attention` endpoint + `Smart Inbox` + `priority` tiers (W507/W508).

# الجزء V — المسارات العلاجية حسب نوع الإعاقة

لكل فئة **بصمة**: تقييمات مميّزة → أهداف → تدخّلات → مقاييس. تُقترَح آلياً عند الفرز.

| ------------------------ | ------------------------------ | -------------------------------------- | -------------------------------------- | ------------------------------------------------- |
| **التوحد (ASD)**         | M-CHAT-R · CARS-2 · Vineland-3 | ABA/DTT · AAC · حمية حسية · خطة سلوك   | معدل الاستجابة المستقلة · إتقان البنود | `DttSession` · AAC · `behavior-plans`             |
| **الشلل الدماغي (CP)**   | GMFCS · MACS · ICF حركي        | علاج طبيعي/وظيفي · جبائر · مقاعد/وضعية | GMFM · مدى الحركة · مستوى GMFCS        | PT/OT · `assistive-device`                        |
| **الإعاقة الذهنية (ID)** | Vineland · تقييم نفسي          | مهارات حياتية · تربية خاصة             | السلوك التكيّفي · استقلالية ADL        | تربية خاصة · `smart-goals`                        |
| **متلازمة داون**         | تطوّري · نطق · سمع/قلب         | تدخّل مبكّر · نطق · علاج طبيعي         | معالم تطوّرية · مفردات                 | تدخّل مبكر · نطق                                  |
| **النطق واللغة**         | تقييمات نطق · بلع              | علاج نطق · AAC · حمية بلع IDDSI        | وضوح الكلام · MLU · مستوى IDDSI        | `therapy-assessments` · AAC · `diet-prescription` |
| **حركية/إصابات**         | ICF حركي · FIM                 | علاج طبيعي · أجهزة مساعدة · تنقّل      | FIM · استقلالية · مسافة مشي            | PT · `assistive-device`                           |
| **حسية (سمع/بصر)**       | فحص سمع/بصر · WHO grade        | تأهيل سمعي/بصري · إحالة                | عتبة السمع · حدة البصر                 | `HearingScreening` · `VisionScreening`            |

> **التوصية:** اجعل المسار يُقترَح آلياً بناءً على فئة الإعاقة (template-driven)، فيُحمّل التقييمات والأهداف والمقاييس كحزمة جاهزة — اتساق سريري + سرعة + امتثال CBAHI.

# الجزء VI — علم المخرجات والقياس

## 6.1 الطبقات الأربع للقياس (في `MeasurementResult`)

1. **خط الأساس (Baseline)** — `applicationNumber:1, purpose:'baseline'`.
2. **الاتجاه (Trend)** — `comparison.trend` (improving/stable/declining).
3. **الدلالة السريرية (MCID)** — `comparison.mcidStatus` ← _الأهم_؛ يمنع الاحتفال بتحسّن وهمي ضمن هامش الخطأ.
4. **بلوغ الهدف (GAS)** — `GasScoring` المربوط بـ `goalId` (سلّم -2…+2).

## 6.2 منهجية GAS الصارمة (Goal Attainment Scaling)

| الدرجة | المعنى                        |
| ------ | ----------------------------- |
| **-2** | الحالة الأساسية (قبل التدخّل) |
| **-1** | تحسّن أقل من المتوقّع         |
| **0**  | المستوى المتوقّع (الهدف)      |
| **+1** | أفضل من المتوقّع              |
| **+2** | أفضل بكثير                    |

ضوابط الجودة: (1) تُكتب مستويات السلّم **قبل** بدء التدخّل لمنع التحيّز؛ (2) موثوقية المقيّمين (inter-rater) عبر مراجعة دورية لعيّنة؛ (3) T-score مجمّع عبر الأهداف للمقارنة المؤسسية.

## 6.3 تصنيف المخرجات (Outcome Taxonomy) على محاور ICF

| المحور      | مثال مخرَج                 | يُقاس بـ               |
| ----------- | -------------------------- | ---------------------- |
| وظائف الجسم | تحسّن مدى الحركة           | تقييمات حركية          |
| النشاط      | استقلالية الأكل/اللبس      | FIM · ADL              |
| المشاركة    | الاندماج المدرسي/الاجتماعي | PedsQL · مؤشرات مشاركة |
| البيئة      | تكيّف المنزل · جهاز مساعد  | تقييم بيئي             |

## 6.4 تجميع المخرجات (Roll-Up)

```text
هدف فردي (GAS/MCID)
   └─► مستفيد (% أهداف محققة)
        └─► برنامج/تخصص (متوسط التحسّن)
             └─► فرع (مخرجات + امتثال)
                  └─► المركز (لوحة تنفيذية + تقرير الجهات)
```

هذا ما يحوّل البيانات الفردية إلى **حجة مؤسسية** للتأمين والجهات التنظيمية والاعتماد.

# الجزء VII — أنطولوجيا الأهداف والترميز السريري

الطبقة التي تجعل النظام _يفهم_ لا _يخزّن_ فقط.

## 7.1 الترميز ثلاثي الطبقات

| النظام        | يصف                                         | الحقل القائم/المقترح                  |
| ------------- | ------------------------------------------- | ------------------------------------- |
| **ICF**       | الأداء الوظيفي (ما يقدر المستفيد على فعله)  | `Goal.icfMapping[].icfCode` ✅ (W452) |

## 7.2 مكتبة أهداف معيارية (Standardized Goal Library)

بدل الأهداف الحرّة (free-text) التي تمنع التجميع، مكتبة أهداف مرمّزة:

ربط: تشخيص → قصور ICF → هدف → تدخّل مبني على دليل → مقياس مخرَج. يتيح استدلالاً مثل: "مستفيدون بنفس قصور ICF استفادوا أكثر من التدخّل X" — أساس التوصية السريرية الذكية (RAG محسّن بالبنية).

# الجزء VIII — حوكمة الذكاء ودورة حياة النماذج

## 8.1 دورة حياة النموذج التنبؤي (ML Lifecycle)

| المرحلة      | المتطلّب                                                                          |
| ------------ | --------------------------------------------------------------------------------- |
| **البيانات** | الخيط الذهبي المكتمل = مصدر الميزات؛ بدون الفجوات الست، الميزات ناقصة             |
| **التدريب**  | هدف واضح (مثلاً: احتمال بلوغ الهدف خلال 90 يوم)؛ مجموعة تدريب/تحقّق/اختبار منفصلة |
| **التقييم**  | دقّة + معايرة (calibration) + ليس فقط AUC                                         |
| **النشر**    | heuristic-first ثم ML اختياري (ADR-011) — لا اعتماد كلّي على صندوق أسود           |
| **المراقبة** | كشف انحراف البيانات (data drift) + إعادة تدريب دورية                              |

## 8.2 العدالة الخوارزمية (Algorithmic Fairness) — حسّاس لفئة هشّة

- **تدقيق التحيّز:** هل يقلّل النموذج تقدير فئة إعاقة معيّنة أو فئة عمرية أو جنس؟ يُدقَّق دورياً.
- **القابلية للتفسير (Explainability):** كل توصية ذكاء تُرفق بـ"لماذا" (الميزات المؤثّرة) — مطلب أخلاقي + CBAHI.
- **حق الاعتراض:** الأخصائي يرفض أي توصية مع سبب مرمّز (`reason-codes.registry.js`)؛ الرفض يغذّي تحسين النموذج.
- **المبدأ الحاكم:** الذكاء _يدعم_ قرار الأخصائي لا _يستبدله_ (1.2.4) — خصوصاً في قرارات مصيرية (تخريج/تصعيد).

## 8.3 حوكمة الخصوصية (PDPL)

كل ميزة ذكاء تحترم: تدرّج الحساسية + TTL ≥30 يوم على بيانات PII + الموافقة الصريحة على الاستخدام الثانوي للبيانات في التدريب.

# الجزء IX — الأسرة وحقوق المستفيد (Co-Production + CRPD)

تأهيل ذوي الإعاقة **شراكة** لا خدمة من طرف واحد.

- **صوت المستفيد إلزامي** في صياغة الأهداف والتخريج (`voice-log` · مؤشر `directPct` = نسبة القرار المباشر مقابل بالوكالة، anti-substitution).
- **القرار المدعوم لا المُستبدِل** (`decision-rights`) — توثيق طبقة الاستقلالية في كل قرار مصيري.
- **تمكين الأسرة:** بوابة ولي الأمر (نتائج، مواعيد، أهداف منزلية) + `caregiver-support` + تسليم منظّم قبل التخريج.
- **المناصرة الذاتية** (`self-advocacy`) — منهج بناء قدرة المستفيد على تمثيل نفسه.

> لوحة المستفيد تُظهر `crpdCompliance` (directPct) بوضوح — حمراء إذا غلب القرار بالوكالة.

# الجزء X — الجودة والسلامة والتحسين المستمر

## 10.1 حلقة التحسين (PDSA) مربوطة بوحدات الجودة

```text
Plan (هدف + مقياس) ─► Do (جلسات) ─► Study (إعادة قياس + MCID) ─► Act (قرار MDT)
        ▲                                                          │
        └──────────────── يغذّي ◄── CAPA / RCA / FMEA ◄────────────┘
```

عند فشل مخرَج أو حدث سلامة → يُفتح `CAPA` (تصحيحي) أو `RCA`/`FMEA` (تحليل جذري). الحلقة **تتعلّم**.

## 10.2 السلامة كطبقة عرضية مستمرة

سقوط · تقرحات · بلع · ألم · حساسية · نوبات → كلها تُغذّي النواة + تولّد تنبيهات حمراء (W1010–W1066) + تفرض تصعيداً عبر Next-Best-Action.

## 10.3 CBAHI كبوابات لا كأرشيف

اربط الـ45 معياراً بأحداث الرحلة: لا تُغلق المرحلة 3 دون توثيق MDT؛ لا تُعتمد خطة دون موافقة مستفيد؛ إلخ.

# الجزء XI — نموذج القيمة الاقتصادية (Value Model)

ما يحوّل النظام من تكلفة إلى استثمار قابل للإثبات.

## 11.1 تكلفة-لكل-مخرَج (Cost-per-Outcome)

```text
تكلفة الحلقة (جلسات × كلفة + موارد)
   ÷ المخرَج المحقّق (أهداف محققة فوق MCID)
   = تكلفة الوحدة من التحسّن السريري
```

## 11.2 العائد على الاستثمار (ROI) للنظام نفسه

| المصدر               | الأثر القابل للقياس                         |
| -------------------- | ------------------------------------------- |
| الإلزام في الواجهة   | تقليل إعادة العمل + اكتمال بيانات أعلى      |
| التنبيهات الاستباقية | تقليل التدهور غير المكتشف → جلسات إنقاذ أقل |
| حزم المسارات         | تقليل زمن إعداد الملف لكل مستفيد            |
| تقارير المخرجات      | تسريع موافقات التأمين + الاعتماد            |

## 11.3 التعاقد القائم على القيمة (Value-Based Contracting)

ربط الدفع التأميني بالمخرجات المثبتة (GAS/MCID) بدل عدد الجلسات — يتطلّب الخيط الذهبي المكتمل كشرط مسبق للمصداقية.

# الجزء XVI — المواصفات التنفيذية (Executable Specifications)

الطبقة (ب): تحويل التصميم إلى عقود قابلة للبناء مباشرة.

## 16.1 عقد البيانات — الفجوة #1 (الهدف ↔ المقياس)

حقل جديد على `Goal` (على نمط `icfMapping` W452):

```jsonc
// Goal.linkedMeasures: [LinkedMeasure]
{
  "measureId": "ObjectId ref:'MeasurementMaster'", // required
  "role": "primary | secondary", // default 'primary'
  "baselineResultId": "ObjectId ref:'MeasurementResult'", // optional (يخدم #5)
  "targetScore": "Number", // القيمة المستهدفة
  "targetDirection": "increase | decrease | maintain", // اتجاه النجاح
  "addedAt": "Date",
  "addedBy": "ObjectId ref:'User'",
}
```

**Invariants** (في `pre('save')`، نفس نمط `icfMapping`):

1. على الأكثر مدخل واحد بـ `role:'primary'`.
2. لا تكرار لـ `measureId` داخل المصفوفة.
3. إذا وُجد `targetScore` فيجب وجود `targetDirection`.

// Session.goalIds: [ObjectId ref:'Goal']   (يحل محل session_goals[] النصية)

| **الاحتفال بتحسّن وهمي**        | قرارات خاطئة        | فرض MCID لا مجرد trend                    |
| **ثقة عمياء بتوصية الذكاء**     | خطر سريري           | اعتماد بشري إلزامي (1.2.4) + قابلية تفسير |
| **بوابة قبول هشّة (enum عربي)** | تلوّث كل ما بعدها   | تطبيع متين (درس W926)                     |
| **النواة قوية والروابط مكسورة** | "ذاكرة بلا عقل"     | إغلاق الفجوات الست                        |
| **تجاهل صوت المستفيد**          | خرق CRPD            | مؤشر directPct إلزامي                     |
| **تقارير وطنية يدوية**          | تأخّر/خطأ امتثال    | محفّزات آلية (الجزء XII)                  |
| **تجاهل التبنّي البشري**        | نظام مثالي مهجور    | إدارة تغيير (الجزء XIII)                  |
| **تحيّز خوارزمي لفئة إعاقة**    | ظلم فئة هشّة        | تدقيق عدالة دوري (الجزء VIII)             |

# الجزء XIX — نموذج النضج ولوحة المؤشرات

## 19.1 نموذج النضج (Maturity Model)

| المستوى            | الوصف              | الحالة                                   |
| ------------------ | ------------------ | ---------------------------------------- |
| 1 أرشيف            | بيانات غير مترابطة | تجاوزته                                  |
| 2 **ذاكرة موحّدة** | سجل زمني واحد      | ✅ **هنا الآن** (النواة من الطراز الأول) |
| 3 خيط ذهبي         | روابط refs كاملة   | الفجوات الست تفصلك                       |
| 4 مدفوع بالمخرجات  | قرارات بـ MCID/GAS | بعد المستوى 3 + الإلزام                  |
| 5 تنبؤي/استباقي    | يقترح قبل التدهور  | الأساس موجود (CDSS)                      |

## 19.2 لوحة المؤشرات المتوازنة (Balanced Scorecard)

| البُعد          | المؤشر الرئيسي                                           |
| --------------- | -------------------------------------------------------- |
| **سريري**       | % أهداف محققة · % فوق MCID · GAS T-score متوسط           |
| **تشغيلي**      | زمن قبول→خطة · جلسات/هدف · معدل الحضور                   |
| **جودة**        | امتثال CBAHI · رايات سلامة مفتوحة · إعادة قياس في موعدها |
| **بيانات**      | % ملفات بخيط ذهبي مكتمل · % أهداف مربوطة بمقياس          |
| **مستفيد/أسرة** | directPct (CRPD) · رضا الأسرة                            |
| **قيمة**        | تكلفة-لكل-مخرَج · ROI                                    |

## 19.3 التدقيقات التشغيلية (للقراءة فقط — تُحوّل المؤشرات إلى أرقام)

مؤشرات بُعد «البيانات» أعلاه ليست تجريدية — لها أدوات تقيسها مباشرة من قاعدة
البيانات دون أي تعديل (آمنة على الإنتاج):

| الأمر                              | يقيس                                                                                                                         | المخرَج                                                                                          |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| `npm run audit:golden-thread`      | **«% ملفات بخيط ذهبي مكتمل»** — يصنّف كل `TherapeuticGoal` بمرحلة انكسار الخيط (بلا مقياس / بلا خط أساس / بلا مخرَج / مكتمل) | درجة صحّة (HEALTHY/PARTIAL/FRAGMENTED) + توصية بأعلى نقطة رافعة. يقبل `--branch=<id>` و`--json`. |
| `npm run audit:goal-consolidation` | أيّ نماذج الأهداف/المقاييس/الخطط تحمل بيانات إنتاج فعلية — يفتح قرارات ADR-040/041/042 التوحيدية                             | حُكم جاهزية لكل ADR. `--json`.                                                                   |

> كلاهما `require.main`-محميّ + منطقه النقي مُختبَر بـ unit tests بلا قاعدة بيانات
> (`golden-thread-integrity-script` 9/9 · `goal-consolidation-readiness-script` 7/7).
> شغّلهما دورياً لرصد انحراف الخيط الذهبي قبل أن يصبح فجوة.
