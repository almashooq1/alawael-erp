> الحالة: مسودة تنفيذية — تنتظر اعتماد الأولويات  
> المالك: فريق المنتج + الهندسة المعمارية

---

## 1. الملخص التنفيذي

يملك المشروع حاليًا **محرك متقدم لإدارة دورة حياة المستفيد** (`backend/intelligence/beneficiary-lifecycle.*`) يدعم 11 حالة و15 انتقالًا مع موافقات متعددة المستويات، MFA، نفاذ، وآثار جانبية (side-effects). لكن هذا المحرك **غير متصل بالنموذج الأساسي للمستفيد** ولا بواجهة المستخدم القديمة، مما يخلق ثغرات حرجة:

- نصف حالات المحرك لا تُقبلها قاعدة البيانات (`Beneficiary.status`).
- المسارات القديمة (`PATCH /api/beneficiaries/:id/status`) تتجاوز المحرك بالكامل.
- معظم الآثار الجانبية لا تزال `deferred` (رسومية فقط).
- الواجهة الأمامية لا تعرف سوى 3 حالات فقط.

**الهدف:** تحويل دورة حياة المستفيد من "محرك معزول" إلى **منظومة متكاملة ذكية وآمنة وقابلة للقياس**، تربط بين النموذج، سير العمل، الواجهات، الإشعارات، والامتثال.

---

## 2. الرؤية: دورة حياة المستفيد الذكية

```
اكتشاف ← تقييم ← قبول ← رعاية نشطة ← انتقال/تخرج ← تواصل مدى الحياة
```

### خصائص الرؤية:

1. **حلقة واحدة للحقيقة (Single Chokepoint):** أي تغيير في حالة المستفيد يمر عبر `beneficiary-lifecycle.service`.
2. **حوكمة آمنة:** موافقات حسب الدور، MFA، Nafath، نافذة إلغاء، وسجل تدقيق غير قابل للتلاعب.
3. **آثار جانبية حقيقية:** إلغاء مواعيد، إغلاق حلقات رعاية، إشعارات فورية، تقارير امتثال.
4. **ذكاء مساعد:** توصيات للمرحلة التالية، تنبيهات المخاطر، توقعات التخرج.
5. **خبرة متصلة:** واجهة موحدة للإداري والعائلة والفريق الطبي.

```
                        │
┌───────────────────────▼─────────────────────────────────────┐
│                    API Gateway                              │
│  /api/v1/beneficiary-lifecycle  (single chokepoint)         │
│  /api/v1/beneficiary-core       (read-only 360)             │

└───────────────────────┬─────────────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────────────┐
│                 Workflow Engine                             │
│  request → approve → execute → cancel/reverse               │

└───────────────────────┬─────────────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────────────┐

                        │

```

## 6. إدارة المخاطر والافتراضات

| المخاطر                                      | الاحتمال | التأثير | التخفيف                                                                |
| -------------------------------------------- | -------- | ------- | ---------------------------------------------------------------------- |
| فشل migration بيانات `inactive` → `archived` | متوسط    | حرج     | تشغيل script ترحيل تدريجي مع نسخ احتياطي                               |
| تعطل الواجهة القديمة بعد إغلاق PATCH status  | عالي     | متوسط   | الإبقاء على endpoint يعيد توجيه إلى lifecycle service مع رسالة توضيحية |
| مقاومة المستخدمين للعملية الجديدة            | متوسط    | متوسط   | تدريب + UI بسيط + رسائل توجيه                                          |
| تداخل MFA/Nafath مع سير العمل                | متوسط    | متوسط   | نافذة MFA واضحة + fallback للعمليات منخفضة الخطورة                     |
| أداء statistics على بيانات كبيرة             | منخفض    | متوسط   | indexes + caching + aggregation pipeline محسّن                         |

### الافتراضات:

- سيتم الاحتفاظ بالمسار `/api/beneficiaries` للقراءة والبحث فقط.
- العمليات الحساسة (transfer, discharge, archive, deletion) تتطلب Nafath في الإنتاج.
- جميع التغييرات الحرجة تُختبر عبر integration tests قبل الدمج.
