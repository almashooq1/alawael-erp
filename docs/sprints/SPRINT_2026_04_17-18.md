### Saudi government adapters (10)

All share the same public contract: `verify / testConnection / getConfig`.

Each adapter supports two modes via env: `{NAME}_MODE=mock|live`. Mock mode is deterministic (see README.md) and requires no network. Live mode calls the real government API and requires credentials.

| GOSI            | Social insurance status             | `GOSI_MODE=live`    |

| Absher / Yakeen | Citizen identity (civil registry)   | `ABSHER_MODE=live`  |

### CI gate

`.github/workflows/sprint-tests.yml` runs on push/PR to `main`/`develop`,
path-scoped to the sprint's file surface. 4 parallel jobs (signer +
adapters + api + seed-dry) with a summary step that fails the workflow
on any regression. Distinct from `ci.yml` which uses `|| echo warning`
— this one **blocks** the PR until green.

## Silent bugs fixed

The new API smoke tests caught **2 production-critical silent bugs** in
the existing route registry:

1. `_registry.js:86` — destructured `{ router }` from `safeRequire`'s
   empty-Router fallback → `undefined` handler → `app.use` threw and
   aborted `mountAllRoutes` → **~15 new admin routes silently never
   mounted** in any environment missing `routes/purchasing.routes.unified`.

2. `_registry.js:111` — same pattern on `inventory.routes.unified`.

Both fixed with defensive destructure:

```js
const x = (safeRequire(...) || {}).router || express.Router();
```

Without the test suite, these would have shipped to staging and
mysteriously 404'd in production until someone stumbled on them.

## Architecture highlights

### Uniform adapter interface

All 10 government adapters expose the same surface:

```ts
interface GovAdapter {
  MODE: 'mock' | 'live';
  verify(params): Promise<{ status, mode, message?, ... }>;
  testConnection(): Promise<{ ok, mode, latencyMs?, error? }>;
  getConfig(): { provider, mode, configured, missing?, timeoutMs };
}
```

Adding a new gov integration (e.g. Tameeni for insurance) is a single
file in `backend/services/` + one line in `gov-integrations.routes.js`.

### Production-grade GOSI

The first adapter was upgraded beyond the mock pattern and serves as
the template for hardening others:

- Token caching with 10s expiry buffer
- Circuit breaker: 5 failures/60s → open for 120s cooldown
- Per-request `AbortController` timeout
- Auto-retry on network error + 401 (forced token refresh)
- Structured logging of circuit state

### ZATCA Phase-2 pipeline

```
Invoice (DB)
  → buildEnvelope() — UUID + ICV + PIH + TLV-QR
  → buildUblInvoice() — UBL 2.1 XML
  → canonicalize() — strip prolog/comments/whitespace
  → digest() — SHA-256 base64
  → SignedInfo → RSA-SHA256 signature via production key
  → injectSignature() — wrap with UBLExtensions + ds:Signature
  → base64 XML → Fatoora API → reporting | clearance
  → response.status → invoice.zatca.zatcaStatus
```

The in-test signature verification (using `crypto.generateKeyPairSync`

- `crypto.createVerify`) **cryptographically proves** the pipeline
  is correct without needing real ZATCA credentials.

## What's NOT done (deliberately)

- **Real vendor onboarding** with GOSI/SCFHS/Fatoora/etc. — business
  step requiring commercial registration numbers, signed agreements,
  and CSR/CSID certificates.
- **App Store / Play Store deployment** — requires Apple + Google
  developer accounts and app bundles not yet signed.
- **Load testing at production scale** — all modules have been unit +
  integration tested; stress testing happens in staging.

Everything marked "live" in this sprint is code-complete; only
environment variables need to change to flip from mock to real.
